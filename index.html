Var "Hoisting"


function foo() {
    bar();
    var x = 1;
}

// same as
function foo() {
    var x;
    bar();
    x = 1;
}
  • Var hoisting is a complex topic that is difficult to understand, especially if you're accustomed to other languages like Java with different scoping.
  • Let's make it easier on everyone and write the code the way it will be interpreted.

Example:


var foo = 1;
function bar() {
    if (!foo) {
        var foo = 10;
    }
    alert(foo); // 10
}
bar();

Combining var Statements


function foo() {
    var x = 1;
    var y = 2;
    var z = 3;
    bar();
}

// might as well just write:
function foo() {
    var x = 1,
        y = 2,
        z = 3;
    bar();
}
  • So we're putting variables at the top.
  • But why stop there?

Rulez!

https://github.com/Wikia/guidelines/blob/master/JavaScript/CodingConventions.md#variables

  • Declare all variables using one var keyword at the top of their scope context. Declaration and assignment on the same line are allowed.

Language Rules

Language rules have an impact on functionality. They were chosen based on performance implications and their tendency to reduce bugs.

JSLint Style Rules

  • Whitespace set up to read like english:
    • Spaces go after commas and colons, not before
    • Spaces after if while for and function as they would in English
    • No space between function name and invoking parentheses
  • Style rules and language rules
  • Modeled rules after English

JSCS

  • JavaScript Code Sniffer
  • Enforces whitespace and style rules
  • Compliments JSHint
  • Config: .jscs.json
  • Ex: "requireCurlyBraces" and "requireSpaceAfterKeywords"
  • Check for redundancies between JSHint and JSCS b/c JSCS doesn't have an "ignore" clause
  • Style checking like:
    • require curly braces for `if`, `else`, `for`, `while`, `do` statements
    • require spaces after keywords: `if`, `else`, `for`, `while`, `function` etc.
  • I suggest you go over these configurations and choose what's best for your oganization and code base

Smelly CSS!

Duplicate Properties


/* bad */
h1 {
    margin: 10px;
    text-transform: uppercase;
    margin: 0; // Second declaration
}

/* acceptable for handling browser support */
.box {
    background: #fff;
    background: rgba(255, 255, 255, .5);
}
  • It's okay to duplicate properties when dealing with various levels of browser support
  • In this example RGBA may not be supported by all browsers

Empty Rule


/* bad */
.cat {
}
  • Broken windows affect
  • Code litter

ID With Extraneous Selector


/* bad */
#submit-button.button {
    ...
}

/* good */
#submit-button {
    ...
}
  • Unclear on the concept - untrustworthy code

Over-specificity & Selector Depth


/* bad - SCSS doesn't have to match the HTML structure! */
.wrapper {
    .wrapper-inner {
        ul {
            li {
                a {
                    .submit-button {
                        ...
                    }
                }
            }
        }
    }
}
.wrapper .wrapper-inner ul li a .submit-button { ... }

Sane Selector Depth


/* good - only do what's necessary */
.wrapper {
    .submit-button {
        ...
    }
}
  • SCSS structure does not have to match the DOM structure
  • Determine what level of selector depth you find to be reasonable for your code base
    • The default is 3
    • We chose 4 to give us breathing room with our legacy
